# Техническое задание на разработку информационной системы для интернет-магазина

# 1. Изучение предметной области

# 1.1. Основные бизнес-процессы интернет-магазина

Прием заказов от клиентов – осуществляется менеджером по продажам. Фиксируется заказ, клиент, список товаров, количество, цена.

Поступление товаров от поставщиков – выполняется кладовщиком. Оформляется приходная накладная с указанием склада, товаров, количества и закупочных цен.

Реализация товаров – осуществляется на основании заказов или напрямую. Отгрузка фиксируется документом реализации. Склад списывает товары, происходит продажа.

Учет остатков – система постоянно отслеживает количество товаров на каждом складе. Автоматически обновляется при поступлениях и реализациях.


# 1.2. Документы, используемые в учете

Заказ покупателя – номер, дата, клиент, менеджер, табличная часть (товары, количество, цена).

Поступление товаров – номер, дата, склад, табличная часть (товары, количество, закупочная цена).

Реализация товаров – номер, дата, клиент, склад, табличная часть (товары, количество, розничная цена).


# 1.3. Пользователи и их роли

Администратор – имеет полный доступ, управляет системой и пользователями.

Менеджер по продажам – работает с клиентами и заказами, оформляет отгрузки.

Кладовщик – регистрирует поступления, контролирует остатки.


# 2. Сбор и структурирование требований

# 2.1. Функциональные требования

1) Ведение справочников: Номенклатура, Клиенты, Склады, Менеджеры.
2) Регистрация документов: Заказ покупателя, Поступление товаров, Реализация товаров.
3) Контроль остатков при оформлении заказов и реализаций.
4) Автоматическое списание товаров со склада при реализации.
5) Поддержка статусов заказов.
6) Формирование отчетов: остатки товаров на складах, продажи за период, движение товаров.
7) Возможность поиска, фильтрации данных в отчетах.


# 2.2. Нефункциональные требования

1) Платформа: 1С:Предприятие 8.3, последняя стабильная версия.
2) Производительность: до 500 товаров, 1000 клиентов, 5000 заказов в год.
3) Интерфейс: удобный, типовой 1С-интерфейс с панелью навигации.
4) Безопасность: разграничение прав доступа по ролям.
5) Соответствие стандартам: код соответствует рекомендациям фирмы 1С.


# 2.3. Приоритеты требований

Критичные: учет заказов, реализация, остатки, справочники, доступ по ролям.

Средний приоритет: отчеты, печатные формы.

Дополнительные: фильтры, поиск, интерфейсная доработка.


# 3. Основные объекты системы

# 3.1. Справочники

1) "Номенклатура"

Назначение: учет товаров. 

Основные реквизиты: артикул, наименование, единица измерения, цены

2) "Клиенты" 

Назначение: учет покупателей.

Основные реквизиты: ФИО/наименование, телефон, email, адрес доставки

3) "Склады"

Назначение: учет мест хранения товаров 

Основные реквизиты: наименование склада

4) "Менеджеры" 

Назначение: Учет сотрудников 

Основные реквизиты: ФИО, должность


# 3.2. Документы

1) "Заказ покупателя"

Назначение: регистрация заказа. 

Основные реквизиты: дата, номер, клиент, менеджер, товары

2) "Поступление товаров" 

Назначение: учет прихода товаров.

Основные реквизиты: дата, номер, склад, товары, закупочная цена

3) "Реализация товаров"

Назначение: учет отгрузки и списания товаров 

Основные реквизиты: дата, номер, клиент, склад, товары, цена


# 3.3. Регистры накопления

1) ОстаткиТоваров – фиксирует количество товара на складе.

2) Продажи – фиксирует факты реализации, продажи по товарам и клиентам.


# 4. Проектирование структуры данных

# 4.1. Реквизиты справочников и документов

Примеры:

Номенклатура: код (артикул), строка (наименование), число (цена), перечисление (единица измерения).

Документы: дата (дата оформления), справочники (ссылки на клиента, менеджера, склад).


# 4.2. Табличные части документов

Включают список товаров: товар (ссылка), количество (число), цена (число).

Общая сумма должна рассчитываться автоматически.


# 5. Бизнес-процессы и сценарии использования

# 5.1. Жизненный цикл документов

1) Поступление товаров: Кладовщик оформляет поступление. Товары добавляются в остатки на складе.
2) Оформление заказа: Менеджер создает заказ. Проверяется наличие товара. Заказу присваивается статус.
3) Реализация товара: Менеджер формирует реализацию (на основании заказа или отдельно). Товары списываются со склада. Документ можно распечатать (накладная, чек).


# 5.2. Правила проверки данных

Нельзя провести заказ, если товаров нет в наличии.

При списании проверяется остаток на складе.


# 5.3. Пользовательские сценарии

Менеджер: оформляет заказы, формирует реализацию, работает со справочниками. Имеет доступ к просмотру справочников, созданию/редактированию заказов и реализаций

Кладовщик: регистрирует поступления, проверяет остатки. Может создавать поступления, просмотривать остатки, без доступа к заказам

Администратор: управляет пользователями, правами и настройками системы. Имеет полные права на все справочники, документы, регистры и настройки
